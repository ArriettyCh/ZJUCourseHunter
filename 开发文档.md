# ZJU Course Hunter 开发文档

## 1. 项目概述

### 1.1 项目目标

为浙大学生提供一个安全、简单的自动抢课工具。用户下载后直接运行，在真实教务界面中手动登录，浏览并选择一门想抢的课程，确认后程序自动循环抢课直到成功。

### 1.2 核心设计理念

| 原则 | 说明 |
|------|------|
| **零配置** | 用户不需要懂技术，不需要手动复制 Cookie 或课程编码 |
| **所见即所得** | 在教务系统真实界面中选课，直观确认目标课程 |
| **不存密码** | 脚本**完全不接触用户密码**，每次运行由用户手动登录，保障账号绝对安全 |
| **单点突破** | 鉴于教务系统频率限制，一次只抢一门课，集中请求频率提高成功率 |
| **会话保活** | 利用抢课请求本身的持续性保持 Session 活跃，无需复杂的自动重连机制（虽然已经做好逆向了，但用不上） |

## 2. 用户使用流程

```
$ python main.py

[自动] 检测环境，安装 Playwright 浏览器（首次运行）
                 ↓
[自动] 弹出浏览器窗口，打开教务网登录页
                 ↓
[用户] 在浏览器中手动完成登录
                 ↓
[自动] 检测到登录成功，向页面注入"开始抢课"按钮（在每门课的操作栏中）
                 ↓
[用户] 浏览课程列表，点击目标课程的"开始抢课"按钮
                 ↓
[自动] 弹出确认框显示课程详情，用户点击"确认开始"
                 ↓
[自动] 浏览器关闭/隐藏，提取 Cookie，进入抢课循环
                 ↓
[自动] 对目标课程持续发送选课请求（保持活跃，防止 Session 过期）
                 ↓
[自动] 抢课成功后发送通知并退出；若遇到不可恢复错误（如被踢下线）则报错停止
```

## 3. 技术架构

### 3.1 技术栈

| 组件 | 选型 | 理由 |
|------|------|------|
| 语言 | Python 3.12 | 生态好，易于维护 |
| 浏览器自动化 | Playwright | 负责登录交互、页面注入与 Cookie 提取 |
| HTTP 请求 | requests/httpx | 负责高频抢课循环，相比浏览器自动化效率更高 |
| 通信机制 | Playwright Binding | `expose_binding` 实现前端 JS 按钮点击直接回调 Python 函数 |

### 3.2 模块划分

```
zju-grab/
├── main.py              # 入口：环境检测 → 启动流程
├── auth.py              # 登录流程
│                        # 1. 启动 Playwright 浏览器
│                        # 2. 等待用户手动登录（监听 URL 变化或特定 Cookie）
│                        # 3. 登录成功后注入 inject.js
├── interface.py         # 交互逻辑 (原 course_picker.py)
│                        # 1. 定义 Python 端回调函数 (handle_grab_request)
│                        # 2. 接收前端传来的课程数据 (xkkh, kcmc 等)
│                        # 3. 弹出确认 CLI/GUI 提示
├── grabber.py           # 抢课核心
│                        # 1. 接收 Cookie 和课程数据
│                        # 2. 构造 requests Session
│                        # 3. 执行死循环请求，处理响应结果
├── inject.js            # 前端注入脚本
│                        # 1. 在教务系统课程列表每一行添加"抢课"按钮
│                        # 2. 点击时提取当前行课程信息，调用 window.py_grab_func
└── requirements.txt     # playwright, requests
```

### 3.3 流程调整说明

1.  **登录安全**：移除了原计划中的密码存储和自动登录功能，规避了密码泄露风险。因为教务系统只在不活跃一段时间以后才会登出，而抢课阶段每秒都在发出请求，教务系统不会登出。（除非用户在其他地方登录了教务系统。但这样就更不能自动重新登录了，否则用户一在别的地方登录，就会触发抢课脚本的自动重新登录，然后用户就被踢下线了）。
2.  **单任务模式**：由于教务系统有频率风控，所以取消购物车设计。用户一次锁定一门课，脚本将以系统允许的最大频率（如 1-3秒/次）针对该课程发送请求。
3.  **异常处理**：
    *   **Session 过期**：由于抢课循环本身就在不断刷新 Session 时间，理论上只要脚本在跑就不会过期。如果因为网络波动导致 Session 失效，脚本直接报错退出，提示用户重新运行登录。
    *   **互斥登录**：如果在抢课过程中用户在手机/其他浏览器登录了教务系统，脚本的 Session 会失效（被踢下线），此时脚本应检测到特定错误码（如 302 跳转或未登录提示）并停止运行。
